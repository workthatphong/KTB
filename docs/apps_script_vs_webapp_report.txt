รายงานเปรียบเทียบการย้ายโปรเจกต์ทั้งหมดไป Google Apps Script

สรุปสั้น
โปรเจกต์นี้ควรอยู่ในรูปแบบ Webapp มากกว่า Apps Script ถ้าย้ายทั้งหมดไป Apps Script จะไม่ใช่แค่ "ช้าลง" แต่จะเจอข้อจำกัดเชิงสถาปัตยกรรม, timeout, quota, ขนาดข้อมูล, และความยืดหยุ่นของ UI ที่ลดลงอย่างมีนัยสำคัญ

ข้อสังเกตสำคัญ
- โปรเจกต์ปัจจุบันเป็น dashboard เต็มรูปแบบ มีหลายหน้า, หลายกราฟ, modal, filter, upload, และ sync กับ Google Sheets/Supabase
- ฝั่ง frontend มีการคำนวณข้อมูลหนักใน Web Worker อยู่แล้ว
- ฝั่ง backend มี API แยกสำหรับ upload, connect/disconnect Google Sheet, sync, delete, และ settings
- มีการกำหนด upload limits และ token guard สำหรับ write endpoints

ข้อมูลอ้างอิงจากตัวโปรเจกต์
- หน้าแอปหลักและ routing: frontend/src/app.tsx
- หน้า Data Management: frontend/src/features/data-management/DataManagementView.tsx
- worker คำนวณข้อมูล: frontend/src/features/dashboard/workers/dashboardWorker.ts
- logic คำนวณ: frontend/src/features/dashboard/workers/workerCalculations.ts
- API: backend/app/presentation/http/routes/api.py
- upload validation: backend/app/presentation/http/uploads.py
- deployment: vercel.json
- README ระบุว่าเป็น Dashboard พร้อม backend, frontend, Supabase, SQLite และ Vercel

ข้อจำกัดจริงของ Google Apps Script ที่เกี่ยวข้อง
- Script runtime: 6 นาทีต่อ execution
- Custom function runtime: 30 วินาทีต่อ execution
- Triggers total runtime: 90 นาที/วัน สำหรับ consumer และ 6 ชั่วโมง/วัน สำหรับ Workspace
- Properties value size: 9 KB ต่อค่า
- Properties total storage: 500 KB ต่อ property store
- URL Fetch calls: 20,000/วัน สำหรับ consumer และ 100,000/วัน สำหรับ Workspace
- Simultaneous executions per user: 30
- Simultaneous executions per script: 1,000
- Web app HTML Service ต้องใช้ doGet() และ HtmlOutput
- HTML Service มีข้อจำกัดบางอย่างของ HTML5 ตามเอกสาร

ส่วนที่คาดว่าจะเกิด regression ถ้าย้ายทั้งหมดไป Apps Script

1) หน้า Dashboard และการเปลี่ยน filter
สถานะปัจจุบัน
- หน้า dashboard คำนวณข้อมูลหลักใน worker ฝั่งผู้ใช้
- มีการ derive ข้อมูล 3 ชุดหลัก: user, system, system second
- มีการ filter และ compute metrics หลายรอบต่อการเปลี่ยน state หนึ่งครั้ง

ผลกระทบเมื่อย้ายไป Apps Script
- การคำนวณต้องข้าม network ไปยัง Apps Script server แทนที่จะรันใน browser worker
- การตอบสนองแต่ละครั้งจะเพิ่ม latency ชัดเจน

ประมาณการ performance drop
- การเปิดหน้า dashboard: ช้าลงประมาณ 2x-4x
- การเปลี่ยน filter / โหมดการแสดงผล: ช้าลงประมาณ 3x-10x
- ถ้าชุดข้อมูลใหญ่และต้องคำนวณซ้ำหลายชุดพร้อมกัน: อาจช้าถึงระดับ 10x ขึ้นไปหรือเริ่ม timeout

เหตุผล
- ปัจจุบัน worker ทำงานใกล้ผู้ใช้และไม่ต้องรอ round-trip จาก server ทุกครั้ง
- Apps Script ต้องรับ request, ประมวลผล, ส่ง response กลับ
- ถ้า UI ต้อง refresh หลายส่วนพร้อมกัน latency จะสะสม

2) การอัปโหลดไฟล์ Excel / CSV
สถานะปัจจุบัน
- โปรเจกต์รับ .csv และ .xlsx
- จำกัด request body, decoded payload, ขนาดต่อไฟล์, และจำนวนไฟล์
- มีการ validate base64 และ size limit ฝั่ง backend

ผลกระทบเมื่อย้ายไป Apps Script
- การรับไฟล์ใหญ่จะชน runtime limit ได้ง่ายกว่า
- ต้องส่งไฟล์ผ่าน HTTP ไปยัง script ซึ่งเพิ่ม overhead
- ถ้าจะเก็บข้อมูลใน Properties หรือใกล้เคียง จะติดเพดานขนาดเก็บข้อมูลเร็วมาก

ประมาณการ performance drop
- ไฟล์เล็ก: ช้าลงประมาณ 2x-5x
- ไฟล์ระดับหลาย MB: ช้าลงประมาณ 5x-10x
- ไฟล์ใหญ่หรือหลายไฟล์พร้อมกัน: เสี่ยง timeout ที่ 6 นาที

ตัวอย่างเชิงตัวเลข
- โปรเจกต์นี้รับไฟล์ได้ถึง 10 MB ต่อไฟล์ และ 25 MB decoded total
- Apps Script Properties เก็บได้ 500 KB ต่อ store และ 9 KB ต่อ value
- แปลว่า storage แบบ Properties เล็กกว่าขอบเขต upload ปัจจุบันประมาณ 95%+ สำหรับกรณีไฟล์ระดับบนสุดที่โปรเจกต์นี้รองรับ

3) การ sync กับ Google Sheets และ storage
สถานะปัจจุบัน
- ใช้ backend + SQLite cache + Supabase sync
- แยก read/write ชัดเจน
- ทำ hydration/sync ได้เป็นขั้นตอน

ผลกระทบเมื่อย้ายไป Apps Script
- ถ้าจะใช้ Google Sheets เป็น storage หลัก จะเจอข้อจำกัดเรื่องขนาด, โครงสร้างข้อมูล, และการอ่านเขียนหลายรอบ
- ถ้าจะใช้ external database ต่อไป ก็เท่ากับยังต้องมี backend อยู่ดี
- การใช้ Apps Script เป็นตัวกลางจะเพิ่ม hop โดยไม่ลดความซับซ้อนจริง

ประมาณการ performance drop
- write/read แบบข้อมูลเล็ก: ช้าลงประมาณ 1.5x-3x
- sync ข้ามหลาย source: ช้าลงประมาณ 3x-6x
- ถ้ามีหลาย action ต่อเนื่องในวันเดียว: เสี่ยงชน URL Fetch quota หรือ trigger runtime quota

4) UI/UX และความยืดหยุ่นของหน้า
สถานะปัจจุบัน
- ใช้ React SPA
- มี route หลายหน้า
- มี lazy load, modal, state แยกหลายชั้น
- มี chart และ timeline ที่ซับซ้อน

ผลกระทบเมื่อย้ายไป Apps Script
- UI จะต้องอยู่ใน HTML Service ซึ่งเหมาะกับ UI แบบเบากว่า
- ทำ SPA ที่ซับซ้อนได้ แต่จะฝืนธรรมชาติมากขึ้น
- การ build, test, deploy, และ debug จะช้าลง

ประมาณการ impact
- ความเร็วในการพัฒนาและแก้บั๊ก: ช้าลงประมาณ 2x-4x
- ความยืดหยุ่นของ UI: ลดลงมาก โดยเฉพาะส่วนที่เป็น route-heavy และ interaction-heavy
- ความเสี่ยง regression จากข้อจำกัดของ HTML Service: สูงกว่าปัจจุบัน

5) การรองรับผู้ใช้หลายคนพร้อมกัน
สถานะปัจจุบัน
- backend แยกออกจาก browser
- scaling ทำได้ง่ายกว่าด้วยการเปลี่ยน infra

ผลกระทบเมื่อย้ายไป Apps Script
- มี simultaneous executions per user อยู่ที่ 30
- มี simultaneous executions per script อยู่ที่ 1,000
- ถ้ามีผู้ใช้หลายคนกด refresh/sync พร้อมกัน จะเริ่มมีคอขวดและ retry behavior ที่เดาได้ยาก

ประมาณการ impact
- กลุ่มผู้ใช้เล็ก 1-5 คน: อาจยังใช้งานได้
- กลุ่มผู้ใช้ระดับทีม 10-30 คน: เริ่มเห็น latency และ error มากขึ้น
- กลุ่มผู้ใช้ระดับองค์กร: เสี่ยงชน quota และการควบคุมโหลดไม่พอ

6) ความปลอดภัยและ permission model
สถานะปัจจุบัน
- มี token guard สำหรับ write endpoints
- แยก env vars และ backend config

ผลกระทบเมื่อย้ายไป Apps Script
- ต้องเลือกว่าให้ script รัน as owner หรือ as user
- permission model จะผูกกับบัญชี Google มากขึ้น
- ถ้าต้องเชื่อม external services หลายตัว จะซับซ้อนขึ้น

ประมาณการ impact
- ความยุ่งยากด้านสิทธิ์และการอนุญาต: เพิ่มขึ้นประมาณ 2x
- ความเสี่ยงเรื่อง permission mismatch และ deployment confusion: เพิ่มขึ้นชัดเจน

ภาพรวม performance loss แบบย่อ
- หน้า dashboard / filter interaction: 2x-10x ช้าลง
- upload / parse / sync: 2x-10x ช้าลง และอาจ timeout
- storage capacity ที่ใช้ได้จริง: ลดลงอย่างมาก ถ้าใช้ Properties เป็นหลัก
- concurrent usage: จำกัดมากกว่าเว็บแอปที่มี backend แยก

ภาพรวม regression ที่น่าจะเจอจริง
- บางงานจะไม่ใช่แค่ช้าลง แต่จะ "ทำไม่ได้เต็มรูปแบบ" เพราะติด runtime / quota / storage
- ถ้าฝืนย้ายทั้งหมด จะต้องลดขนาดข้อมูล ลดความซับซ้อนของ UI และลดความถี่ของ sync
- ถ้าจะรักษาความสามารถเดิมไว้ ต้องมี backend ภายนอกอยู่ดี ซึ่งทำให้ข้อดีของ Apps Script ลดลงมาก

ข้อสรุปสุดท้าย
- ถ้าต้องการ dashboard แบบนี้ให้ทำงานครบทุกฟีเจอร์และคุม performance ได้: Webapp เหมาะกว่า
- Apps Script เหมาะเป็นตัวช่วยเสริมใน Google Workspace มากกว่า ไม่เหมาะเป็นแกนหลักของระบบนี้
- ถ้าจะย้ายทั้งหมดไป Apps Script ควรยอมรับล่วงหน้าว่าจะมี regression ทั้งด้านความเร็ว, ความจุ, concurrency, และความยืดหยุ่นของ UI

หมายเหตุ
- ตัวเลขเปอร์เซ็นต์ในรายงานนี้เป็นการประมาณจากสถาปัตยกรรมของโปรเจกต์และข้อจำกัดที่ Google เผยแพร่ ไม่ใช่ benchmark วัดจริงของโค้ดชุดนี้
- หากต้องการตัวเลขที่แม่นยำระดับ production ควรทำ benchmark เปรียบเทียบจริงบนชุดข้อมูลตัวอย่างขนาดเล็ก, กลาง, และใหญ่
